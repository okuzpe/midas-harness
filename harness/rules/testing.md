# Rule: Testing (always-on)

Testing rules apply from Phase 7 (Sprint Execution) onward and are re-audited each Phase 8.
Every sprint's Definition of Done references this file. Stack-specific extensions (test runner
config, coverage thresholds, E2E tooling) are generated by `/define-conventions`.

> **Every item carries a `**CHECK:**`** — the concrete condition the Phase-8 audit evaluates: a
> command/grep where one exists, or a `manual:` observable when judgment is required (the auditor
> records pass/fail with the test name or `file:line` as evidence).

## Checklist

### Coverage contract
- [ ] Every behavior change (new feature, bug fix, refactor with observable effect) ships with
      at least one new or updated test.
      **CHECK:** `manual:` the sprint diff pairs each behavioural change with a new/updated test in the same range; a behaviour change with no test delta is a fail.
- [ ] Tests assert observable behavior, not internal implementation details — restructuring
      internals must not require test rewrites unless behavior changes.
      **CHECK:** `manual:` tests assert public outputs/effects; a test reaching into private state/mocks-everything is a fail.
- [ ] The test suite passes in full before the sprint task is marked done (`stage_status: passed`).
      **CHECK:** the project test command (`npm test` / `pytest` / …) exits 0 with zero failures.

### Test types and placement
- [ ] Unit tests cover each discrete function/module in isolation; dependencies are stubbed/mocked.
      **CHECK:** `manual:` each new public function/module has a unit test with its dependencies stubbed.
- [ ] Integration tests cover at least one critical path through each module boundary defined
      in `product/architecture.md`.
      **CHECK:** `manual:` at least one integration test exercises each architecture module boundary touched this sprint.
- [ ] End-to-end tests (if the stack includes a UI or API surface) cover the primary user
      journeys listed in `product/sprints/NN-*.md` acceptance criteria.
      **CHECK:** `manual:` each acceptance-criterion journey has an E2E/API test (or a `/midas-verify` record); an uncovered journey is a fail.
- [ ] Test files live adjacent to the code they test or in a mirror directory —
      `/define-conventions` pins the exact convention per stack.
      **CHECK:** every test file sits in the pinned location (adjacent or mirrored); a stray test path is a fail.

### Test quality
- [ ] Each test has one assertion focus; compound tests that can fail for multiple unrelated
      reasons are split.
      **CHECK:** `manual:` each test targets one behaviour; a test asserting several unrelated outcomes is a fail.
- [ ] Test names state the scenario and expected outcome (not just the function name).
      Pattern: `<unit> <scenario> <expected>` or BDD `given / when / then`.
      **CHECK:** `manual:` test titles name scenario + expected result; a title that is just the function name is a fail.
- [ ] No tests are marked skip/xfail/pending without a linked issue and expiry condition.
      **CHECK:** `grep -rnE "\.(skip|only)|xit\(|xfail|@pytest.mark.skip|test.todo" <tests>` → each match carries a linked issue + expiry, else fail.
- [ ] No assertions that trivially pass (e.g. `assert True`, `expect(1).toBe(1)`) with no
      real behavior check.
      **CHECK:** `grep -rnE "assert\s+True|expect\((true|1)\)\.toBe\(\1\)|assert 1 == 1" <tests>` → empty.

### Isolation and determinism
- [ ] Tests do not depend on execution order; each test sets up and tears down its own state.
      **CHECK:** the suite passes when run in random/sharded order; shared mutable state across tests is a fail.
- [ ] Tests do not make live network calls in CI — external services are mocked or stubbed
      (contract tests are acceptable and must be tagged as such).
      **CHECK:** `grep -rnE "https?://(?!localhost|127\.0\.0\.1)" <tests>` → empty, or each match is a tagged contract test.
- [ ] Tests do not write to shared filesystem paths without cleanup; tmp directories are used
      and removed.
      **CHECK:** `manual:` filesystem-touching tests use a tmp dir and clean up in teardown.
- [ ] Randomness is seeded and recorded; time-dependent tests use an injectable clock.
      **CHECK:** `grep -rnE "Math.random|Date.now\(\)|new Date\(\)" <tests>` reviewed; unseeded randomness or a real clock in a time-dependent test is a fail.

### CI gate
- [ ] The full test suite runs automatically on every PR/push via the CI workflow.
      **CHECK:** the CI workflow (`.github/workflows/*`) runs the test command on push/PR; absent, it is a fail.
- [ ] A failing test blocks merge — no bypassing test failures to ship faster.
      **CHECK:** `manual:` branch protection / required check makes the test job mandatory for merge.
- [ ] Flaky tests are tracked in a dedicated issue and fixed or quarantined within one sprint.
      **CHECK:** `manual:` any known-flaky test has a tracking issue and a fix/quarantine within the sprint.
