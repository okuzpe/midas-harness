# Playbook: Add a Drizzle migration

| Field | Value |
|---|---|
| **Use when** | Adding or changing a table/column in the schema |
| **Stack** | drizzle-orm@0.40.x + drizzle-kit · PostgreSQL 17 — Context7-verified |
| **Owner tier** | build |

## Steps

1. Edit `src/lib/db/schema.ts` (the single source of truth for table shapes). A new tenant-scoped table
   **must carry a `workspaceId` FK** — that is the project's non-obvious invariant.
2. Generate the migration with the project's drizzle-kit generate script (e.g. `npm run db:generate` if
   defined) and **review the emitted SQL** under `drizzle/migrations/` before committing — never
   hand-write it.
3. Keep the change **additive** (no destructive `ALTER`/`DROP` within a single sprint; stage a type change
   across migrations) — see the sprint's risk note.
4. Update `schema.test.ts`: assert the new column/enum exists and, for a tenant table, that
   `workspaceId` is present (the tenant-isolation guard test).

## Must honor

- **Rules:** `workspaceId` on every tenant table (`harness/rules/security.md`); DB columns snake_case ↔
  TS fields camelCase, and pinned deps (`harness/rules/naming.md`, `harness/rules/code-quality.md`).
- **Context7:** fetch `drizzle-orm@0.40.x` "schema definition + migrations" and the drizzle-kit CLI usage
  before editing — APIs differ across minor versions (`harness/rules/context7-usage.md`).

## Done when

- [ ] Migration generated by drizzle-kit (not hand-written) and the additive SQL reviewed in the diff.
- [ ] `schema.test.ts` updated and passing; a new tenant table has a `workspaceId` column.
- [ ] A clean apply on a fresh Postgres is exercised by CI (per `product/architecture.md`).
- [ ] All applicable `harness/rules/*` still pass (audited in Phase 8).
