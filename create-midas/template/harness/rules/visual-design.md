# Rule: Visual design fundamentals (always-on)

Applies to any product that ships a **user-facing UI** (web, mobile, desktop). It is the
non-negotiable quality floor for **hierarchy, typography discipline, emphasis, spacing intent, and
lightweight UX** — what keeps AI-generated UI from looking "amateur" (everything the same weight,
arbitrary type, competing CTAs, missing async states). Stack-specific extensions (the component
library's variant API, layout primitives) are generated by `/define-conventions`.
**Headless / backend-only / CLI project?** Every CHECK here is **N/A** — record each one as
`N/A (no UI)` in the audit; do not delete the rule.

> **Delegation — do not duplicate `accessibility.md`.** WCAG contrast, visible focus, reduced-motion,
> text alternatives, min target size, containment/overflow, and z-index are governed exclusively by
> [`accessibility.md`](./accessibility.md). This rule covers **visual hierarchy and composition** only.
> Component-level styling patterns live in [`../design-system/components.md`](../design-system/components.md).

> **Every item carries a `**CHECK:**`** — the concrete condition the Phase-8 audit evaluates: a
> command/grep where one exists, or a `manual:` observable when judgment is required (the auditor
> records pass/fail with `file:line` or the screen/component name as evidence). The greps are
> illustrative, portable POSIX-ERE commands a human auditor runs — adapt them to the stack's file
> extensions and component primitives. `<ui-src>` is the UI source root from the folder-structure rule
> that `/define-conventions` emits.

## Checklist

### Visual hierarchy (what matters most)
- [ ] Each view/screen has **one** primary call-to-action (filled `--ds-action-*` button or equivalent).
      Secondary actions use secondary/ghost variants — never multiple competing primaries.
      **CHECK:** `manual:` on each key screen, exactly one primary CTA is visually dominant; a second
      filled primary on the same view is a fail.
- [ ] Heading levels descend without skips (`h1` → `h2` → `h3`, not `h1` → `h4`).
      **CHECK:** `grep -rniE "<h[1-6]" <ui-src>` → `manual:` each page's heading order has no skipped
      levels; a decorative size bump without semantic level is a fail.
- [ ] Title > subtitle > body > metadata is obvious at a glance (size, weight, and colour from tokens —
      not everything the same emphasis).
      **CHECK:** `manual:` the primary heading uses `--ds-text-2xl`/`--ds-text-3xl` or larger; body copy
      uses `--ds-text-md`; metadata/labels use `--ds-text-sm` or `--ds-text-secondary` — a screen where
      all text reads the same size/weight is a fail.

### Typography discipline
- [ ] At most **two** font families in product UI: sans (`--ds-font-sans`) and mono (`--ds-font-mono`
      for code only).
      **CHECK:** `grep -rniE "font-family" <ui-src>` → every declaration is `var(--ds-font-sans)` or
      `var(--ds-font-mono)` (or inherits from the base reset); a third family or a raw stack name is a fail.
- [ ] Font sizes come from the modular scale (`--ds-text-*`), not arbitrary values.
      **CHECK:** `grep -rniE "font-size:[[:space:]]*[0-9.]+(px|rem|em)" <ui-src>` → each hit is
      `var(--ds-text-*)` or lives in the token definition file; a raw size in component code is a fail.
- [ ] Line heights come from tokens (`--ds-leading-*`), not magic numbers (except the base html reset).
      **CHECK:** `grep -rniE "line-height:[[:space:]]*[0-9.]+" <ui-src>` → each hit in component code is
      `var(--ds-leading-*)`; a raw unitless/px line-height in a component is a fail.

### Spacing and grid intent
- [ ] Padding, margin, and gap use `--ds-space-*` tokens (8 px grid). Proximity groups related elements;
      section breaks use larger steps (`--ds-space-8`+).
      **CHECK:** `manual:` related controls share `--ds-space-2`/`--ds-space-4` gaps; unrelated sections
      are separated by ≥ `--ds-space-6`; arbitrary px gaps in component CSS are a fail (see
      `components.md` cross-cutting rule 6).

### Emphasis and colour restraint
- [ ] One brand accent drives primary actions; intent colours (success/warning/danger/info) are reserved
      for status — not decoration.
      **CHECK:** `manual:` the primary CTA uses `--ds-action-*`; status badges/toasts use intent tokens;
      a decorative rainbow of accent colours on one screen is a fail. Hardcoded colour values are graded
      under `accessibility.md` (do not re-audit here).
- [ ] The primary CTA is visually distinct from secondary actions on the same view.
      **CHECK:** `manual:` primary vs secondary/ghost variants are distinguishable without relying on
      colour alone (weight, fill vs outline, or position).

### Lightweight UX (clarity, not research)
- [ ] Each key view states the **primary action** clearly (what happens next is obvious without reading
      everything).
      **CHECK:** `manual:` a first-time reviewer can name the primary action within 5 seconds on each
      key screen; buried or ambiguous primary action is a fail.
- [ ] Reading layout respects an F- or Z-pattern on marketing/landing views; app views lead with the
      task title and primary action (not buried below the fold on desktop).
      **CHECK:** `manual:` on landing/marketing pages, headline + primary CTA appear in the first viewport;
      on app views, the page title and primary action precede dense secondary content.
- [ ] Views that load data asynchronously ship **empty**, **loading**, and **error** states — not a blank
      shell or silent failure.
      **CHECK:** `manual:` each data-dependent view has identifiable empty/loading/error UI (component
      name or route cited in the audit); a spinner-only forever or empty white box is a fail.
