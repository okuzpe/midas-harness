# Rule: Naming (always-on)

Consistent names are the cheapest form of documentation. These rules apply from Phase 5 onward
and are audited in Phase 8 by grep-scanning new files and symbols in the sprint diff.
Stack-specific casing conventions (e.g. snake_case for Python, camelCase for JS/TS) are
generated by `/define-conventions` and extend, not override, the language-agnostic rules here.

> **Every item carries a `**CHECK:**`** — the concrete condition the Phase-8 audit evaluates: a
> grep/command where one exists, or a `manual:` observable when judgment is required (the auditor
> records pass/fail with `file:line` evidence).

## Checklist

### Files and directories
- [ ] All file and directory names use `kebab-case` with no spaces.
      **CHECK:** `git diff --name-only <base>..HEAD` shows no path segment matching `[A-Z _]` (outside framework-mandated names like `README`, `Dockerfile`).
- [ ] File names describe their primary export or responsibility (`user-auth.ts`, `invoice-service.go`),
      not a vague container (`utils.ts`, `helpers.py`).
      **CHECK:** `git diff --name-only | grep -iE "/(utils?|helpers?|misc|common|stuff)\.[a-z]+$"` → empty (or each justified).
- [ ] Index/barrel files exist only when the directory represents a public module boundary;
      they re-export only the public surface, not internal details.
      **CHECK:** `manual:` each `index.*` barrel sits on a public module boundary and re-exports only the public surface.
- [ ] Test files follow the pattern `/define-conventions` pins (typically `*.test.*` or `*.spec.*`
      adjacent to the source, or mirrored under a `tests/` directory).
      **CHECK:** every test file matches the pinned pattern (`git diff --name-only | grep -iE "test|spec"` all conform); a misnamed test is a fail.

### Types, classes, and components
- [ ] Types, classes, interfaces, and UI components use `PascalCase`.
      **CHECK:** `grep -rnE "(class|interface|type|enum)\s+[a-z]" src/` → empty (declarations start uppercase).
- [ ] Names describe *what the thing is*, not *how it is implemented*
      (`InvoiceRepository`, not `InvoiceClass` or `InvoiceObject`).
      **CHECK:** `grep -rnE "(Class|Object|Impl|Manager|Data)\b" src/` reviewed; an implementation-noise suffix that adds no meaning is a fail.
- [ ] Abstract or base types carry a meaningful qualifier, not a generic suffix
      (`BaseRepository` is acceptable; `AbstractThing` is not).
      **CHECK:** `manual:` abstract/base type names carry a domain qualifier; a generic `AbstractThing`/`BaseObject` is a fail.

### Functions and methods
- [ ] Functions and methods use the casing convention set by `/define-conventions` per language.
      **CHECK:** grep new function declarations against the stack casing rule; a casing mismatch is a fail.
- [ ] Function names begin with a verb that describes the effect or query:
      `createUser`, `fetchInvoice`, `isExpired`, `hasPermission`.
      **CHECK:** `manual:` each new function name starts with a verb/query word; a noun-only function name is a fail.
- [ ] Boolean-returning functions use `is`, `has`, `can`, or `should` prefixes.
      **CHECK:** `manual:` functions/methods returning boolean use an `is/has/can/should` prefix.
- [ ] Event handlers use `handle` or `on` prefixes (`handleSubmit`, `onUserCreated`).
      **CHECK:** `manual:` handler functions use a `handle`/`on` prefix; a bare handler name is a fail.

### Variables and constants
- [ ] Variable names describe intent, not type: `activeUsers`, not `userArray` or `arr`.
      **CHECK:** `grep -rnE "\b\w+(Array|List|Obj|Str|Num|Map)\b\s*=" src/` reviewed; a type-suffixed variable name is a fail.
- [ ] Single-letter variables are allowed only for loop indices (`i`, `j`) and conventional
      math/algorithm notation; all other variables use full descriptive names.
      **CHECK:** `manual:` single-letter names appear only as loop indices or standard math notation.
- [ ] Constants (immutable values used in multiple places) are named in
      `SCREAMING_SNAKE_CASE` or the language-idiomatic equivalent (set by `/define-conventions`).
      **CHECK:** `manual:` shared immutable constants use the pinned constant casing; a lowercase shared constant is a fail.
- [ ] Abbreviations are avoided unless they are industry-standard (`URL`, `ID`, `API`, `HTTP`).
      **CHECK:** `manual:` non-standard abbreviations (e.g. `usr`, `cfg`, `tmp` as identifiers) are fails; standard ones are allowed.

### Consistency
- [ ] The same concept uses exactly one name throughout the codebase — no synonyms
      (`user` vs `account` vs `member` for the same entity).
      **CHECK:** `manual:` grep the synonyms for one entity (`grep -rinE "user|account|member" src/`); two names for the same concept is a fail.
- [ ] Glossary terms from `product/idea.md` and `product/architecture.md` are used verbatim
      in code; the code is the authoritative implementation of the product vocabulary.
      **CHECK:** `manual:` each domain noun in code matches a glossary term from `product/idea.md` / `product/architecture.md`.
- [ ] When renaming a concept, all occurrences are updated in one commit, and the change is
      documented in an ADR if it crosses a public API boundary.
      **CHECK:** `manual:` a rename touches all occurrences in one commit; a partial rename leaving the old name is a fail.
