# Rule: Security (always-on)

These rules are non-negotiable from Phase 5 onward. The Phase-8 audit treats any unchecked item
here as a **gate blocker** unless a conscious exception is recorded in the sprint's ADR log.
Stack-specific extensions (e.g. OWASP Top 10 for web, container hardening for cloud) are
generated by `/define-conventions` and appended under `harness/rules/`.

> **Every item carries a `**CHECK:**`** — the concrete condition the Phase-8 audit evaluates: a
> grep/command where one exists, or a `manual:` observable when judgment is required (the auditor
> records pass/fail with `file:line` evidence). A failed security CHECK blocks the gate unless a
> conscious exception is logged in an ADR.

## Checklist

### Secrets and credentials
- [ ] No secrets, tokens, API keys, or passwords appear in any committed file — including
      config files, test fixtures, seed data, and documentation examples.
      **CHECK:** `git grep -nE "(sk-[A-Za-z0-9]{16,}|ghp_[A-Za-z0-9]{16,}|-----BEGIN [A-Z ]*PRIVATE KEY)"` → empty; a secret-scanner (gitleaks/trufflehog) on the diff finds nothing.
- [ ] All secrets are read from environment variables or a per-developer local config that is
      listed in `.gitignore`.
      **CHECK:** `manual:` every credential the code consumes resolves from `process.env`/env equivalent or a gitignored local file; a hardcoded credential is a fail.
- [ ] MCP config files (`.mcp.json`) use `${ENV_VAR}` placeholders, never literal values.
      **CHECK:** `grep -nE "(token|api[_-]?key|secret|password)\"\s*:\s*\"[^$]" .mcp.json` → empty (matches `/midas-doctor`'s `mcp:secret-free` check).
- [ ] `.gitignore` explicitly excludes all known secret-bearing file patterns
      (`.env`, `*.pem`, `*secret*`, `*credential*`).
      **CHECK:** `grep -E "\.env|\*\.pem|secret|credential" .gitignore` matches each pattern.

### Least privilege
- [ ] Filesystem MCP scopes are restricted to the minimum directories the agent actually needs.
      **CHECK:** `manual:` review `.mcp.json` filesystem args; a scope broader than the project working dirs is a fail.
- [ ] Git MCP scopes disallow force-push and branch-deletion unless explicitly required and
      documented in an ADR.
      **CHECK:** `manual:` git MCP config grants no force-push/branch-delete without a referencing ADR.
- [ ] Third-party API tokens are scoped to the minimum permissions required (read-only where
      write is not needed).
      **CHECK:** `manual:` each token's documented scope matches its actual use; a write/admin token used only for reads is a fail.
- [ ] Service accounts and CI credentials carry only the roles required by that pipeline step.
      **CHECK:** `manual:` CI workflow permissions block (e.g. `permissions:` in the workflow) grants only the steps' required scopes; default-broad tokens are a fail.

### Input validation and output encoding
- [ ] All external inputs (HTTP requests, CLI args, file uploads, webhooks, environment values)
      are treated as untrusted and validated at the boundary.
      **CHECK:** `manual:` each boundary validates/parses input (schema validator, type guard) before use; an unvalidated path is a fail (evidence: `file:line`).
- [ ] User-supplied strings are never interpolated directly into shell commands, SQL queries,
      or template strings — use parameterized queries and safe APIs.
      **CHECK:** `grep -rnE "(exec|spawn)\(.*\$\{|query\(\s*[\"'\`].*\$\{|\+ req\.(body|query|params)" src/` → empty (string-built SQL/shell from user input is a fail).
- [ ] Output to the DOM / templates is encoded appropriately (HTML-escape, JSON-encode, etc.)
      to prevent injection; no `innerHTML` or equivalent with untrusted data.
      **CHECK:** `grep -rnE "innerHTML|dangerouslySetInnerHTML|v-html|\|\s*safe" src/` → empty, or each match proven to use sanitized/constant data.

### Dependencies
- [ ] Before adding a dependency, confirm it has no known high/critical CVEs
      (run `npm audit` / `pip-audit` / equivalent and document the result in the PR).
      **CHECK:** `npm audit --audit-level=high` (or `pip-audit`) exits clean on the new deps; the PR records the result.
- [ ] Dependencies are pinned to exact versions (see `code-quality.md`); lock files are committed.
      **CHECK:** lockfile present and committed in the diff; manifest has no unbound ranges (see `code-quality.md` pinning CHECK).
- [ ] Transitive dependencies are not silently upgraded — lock file diffs are reviewed in PR.
      **CHECK:** `manual:` lockfile diff in the PR is reviewed; an unexplained transitive bump is a fail.

### Data exposure
- [ ] Error messages returned to callers contain no internal paths, stack traces, or secret values.
      **CHECK:** `manual:` error responses return a safe message/code; a raw stack trace or path reaching the client is a fail.
- [ ] Logs never record secrets, PII, or session tokens (mask or omit before writing).
      **CHECK:** `grep -rnE "log.*(password|token|secret|ssn|email)" src/` → reviewed; logging a raw secret/PII value is a fail.
- [ ] Network responses do not leak implementation details (internal IDs, framework fingerprints,
      server version headers) beyond what the product spec requires.
      **CHECK:** `manual:` response headers/bodies expose no server version or internal IDs not required by spec (e.g. `X-Powered-By` disabled).

### Transport and storage
- [ ] All network communication with external services uses TLS; no `http://` in production
      endpoints (enforced by config, not just convention).
      **CHECK:** `grep -rnE "http://(?!localhost|127\.0\.0\.1)" src/ config/` → empty.
- [ ] Sensitive data at rest is encrypted if the product spec requires it; the choice is
      recorded in `product/architecture.md`.
      **CHECK:** `manual:` if the spec requires encryption-at-rest, `product/architecture.md` records the mechanism and the code/infra applies it.
